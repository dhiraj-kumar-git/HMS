# HMS Project Rules & Guidelines

These rules must be followed for all code changes in this application.

## 1. Test-Driven Development (TDD)
- Always follow a Test-Driven Development (TDD) approach to prevent regression failures.
- Validate that the new code changes are working as expected.
- Ensure high unit testing coverage for all new code changes.

## 2. Documentation Maintenance
- Continuously update the `README.md` and `TESTING.md` files based on the latest project structure and changes.
- If there are any changes to the backend API, ensure they are reflected in the Swagger documentation.
- Update the `API_TESTING_GUIDE.md` file regularly when API endpoints or testing workflows change.

## 3. Reference Guides
- Maintain the `GUIDE.md` file in the parent folder. This acts as a complete guide to a new user trying to use the application, containing detailed information on all available functionalities, pages, and dashboards. Keep it updated with every new code change.
- Ensure the application is well-documented both technically and functionally:
  - **Technical Documentation:** Must be detailed enough to help a junior intern get onboarded to the application.
  - **Functional Documentation:** Must help the user understand the functionalities and use cases of this project application.

## 4. Tracking Changes
- Maintain the `NEW_CHANGES.md` file in the parent folder. This file keeps track of a summary of all the changes done to the project application before committing and pushing the code changes. Update it whenever any new code changes take place.
